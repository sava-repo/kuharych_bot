# Proposal: Пользовательские категории рецептов

## Why

Категории рецептов захардкожены в три значения (`завтрак / основное блюдо / десерт`) и влиты в идентичность рецепта: `recipes PK = (category, slug)`, `reel_index reel_id → (category, slug)`, `recipe_ingredients PK = (category, slug, ingredient)`. Из-за этого:

- Пользователь не может завести свои категории («мамины рецепты», «на праздник», «супы»).
- Один и тот же рецепт не может лежать в разных категориях у разных пользователей: `reel_index` держит `reel_id → (category, slug)` глобально, а `process_video` при дублирующем рилсе переиспользует чужую категорию.
- Перемещение рецепта между категориями (`group_manager.move_recipe_category`) делает `UPDATE ... WHERE category = ? AND slug = ?` без фильтра по группе — меняет категорию рецепта во ВСЕХ группах одновременно.

Нужны персональные таксономии: категории принадлежат группе (личная группа `pers_{id}` выступает личным набором пользователя), а категория рецепта — это тег членства в группе, а не часть идентичности рецепта.

## What Changes

- **Новая сущность `group_categories`** (category_id, group_id, name, position, is_default) с CRUD: добавить / переименовать / удалить (MVP).
- **Рецепт отвязан от категории**: `recipes` получает суррогатный `recipe_id`; `recipe_ingredients`, `reel_index`, `group_recipes` ссылаются на `recipe_id`.
- **Категория = тег группы**: `group_recipes` хранит `category_id` для `(group_id, recipe_id)`; PK `(group_id, recipe_id)` — одна категория на рецепт в группе.
- **LLM-промпт динамический**: список категорий активной группы подставляется в `prompts/system.txt`; LLM выбирает одну, при неуверенности — категория с `is_default=1`.
- **Сидирование**: личные (`pers_*`) и кастомные (`grp_*`) группы при создании получают 3 дефолтные категории; `основное блюдо` = `is_default=1`.
- **Дубль-рилс** (контент уже распознан в другой группе) добавляется в новую группу под дефолтную категорию — без повторного LLM-вызова.
- **Изоляция перемещения**: `move_recipe_in_group(group_id, recipe_id, new_category_id)` обновляет ровно одну строку `group_recipes`.

## Capabilities

### Added Capabilities
- `category-management`: управление категориями уровня группы и отвязка категории от идентичности рецепта.

## Impact

- **Схема БД**: table-rebuild миграция в `services/database.py` (SQLite не умеет менять PK через ALTER); обязателен бэкап `data/bot.db`.
- **Сервисы**: `group_manager.py` (CRUD категорий + все recipe-функции под `recipe_id`), `recipe_parser.py` (динамический промпт), `recipe_pipeline.py`, `cache.py`, `rotation.py`.
- **Хендлеры**: `menu.py`, `buttons.py`, `keyboards.py`, `groups.py` (сидирование при создании группы) + новый UI категорий.
- **Модели/константы**: `models/recipe.py` (категория уходит из идентичности и из `format_message`), `constants.py` (`VALID_CATEGORIES` → `DEFAULT_CATEGORIES`).
- **Dashboard**: `pages/3_🍳_Рецепты.py` и смежные — запросы через `recipe_id`.
- **Тесты**: расширение `tests/unit/test_db_schema.py` (миграция), `test_parser.py`, `test_buttons.py`, `test_menu.py`.

## Non-goals

- Порядок категорий в меню (reorder-кнопки) — отложен; колонка `position` заложена про запас.
- Произвольное число категорий на один рецепт у одного пользователя (многие-ко-многим) — остаётся «одна категория на рецепт в группе».
- FTS5 / замена поиска — меняется только JOIN-структура, механика поиска прежняя.
- Перенос пользовательских категорий между группами (экспорт/импорт).
