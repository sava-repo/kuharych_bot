## Context

Бот «Кухарыч» — Telegram-бот для управления рецептами. Рецепты хранятся в GitHub (Markdown), метаданные — в SQLite. Пользователи запрашивают случайный рецепт через кнопки меню (завтрак / основное блюдо / десерт) или кнопку «🎲 Другой рецепт».

Текущая реализация (`handlers/menu.py`) использует `random.shuffle()` на полном списке slug'ов группы → категории. Повторы не отслеживаются.

## Goals / Non-Goals

**Goals:**
- Исключать недавно показанные рецепты из случайного выбора (per-user, per-category)
- Гарантировать, что при наличии ≥2 рецептов пользователь не увидит повтор подряд
- Прозрачно сбрасывать rotation, когда все рецепты категории были показаны
- Работать как для кнопок категорий, так и для поиска по ингредиентам

**Non-Goals:**
- Персистентное хранение rotation (не нужно — потеря при рестарте незаметна)
- Rotation между разными группами одного пользователя (независимые контексты)
- Настраиваемый пользователем размер rotation
- Rotation для других типов контента

## Decisions

### 1. Хранилище: In-memory dict + deque

**Выбор:** `dict[tuple[int, str], deque[str]]` — ключ `(user_id, category)`, значение — deque slug'ов.

**Альтернативы:**
- SQLite-таблица: персистентно, но избыточно для UX-фичи, добавляет I/O на каждый запрос
- Redis: overkill для single-instance бота

**Обоснование:** In-memory дешевле всего по сложности. Deque автоматически вытесняет старые элементы при `maxlen`. Данные теряются при рестарте — но пользователь просто увидит один знакомый рецепт после рестарта, это приемлемо.

### 2. Размер deque: динамический (total - 1)

**Выбор:** `maxlen = max(len(all_slugs) - 1, 0)` — при каждом запросе вычисляется заново.

**Обоснование:** Гарантирует, что минимум 1 рецепт всегда доступен для показа. Не нужен отдельный fallback-код для случая «все рецепты в rotation». Если в категории 1 рецепт — deque будет пустой, повтор допустим.

### 3. Scope: (user_id, category)

**Выбор:** Rotation привязан к пользователю и категории, без учёта group_id.

**Обоснование:** Если пользователь переключает группу, список slug'ов всё равно другой (запрос из БД по group_id). Старые slug'и из rotation естественным образом не найдутся в новом списке и будут проигнорированы. Нет нужды усложнять ключ.

### 4. Интеграция: параметр `exclude` в функции выбора

**Выбор:** Добавить `exclude: list[str]` в `_try_get_random_recipe` и `_try_get_recipe_from_slugs`. Фильтрация slug'ов до `random.shuffle`.

**Альтернативы:**
- Фильтрация внутри rotation-модуля: нарушает SRP, rotation не должен знать про GitHub
- Фильтрация в хэндлере до вызова: дублирование логики в 4 хэндлерах

**Обоснование:** Параметр `exclude` — чистая функция фильтрации, минимальное изменение сигнатуры. Хэндлер отвечает за получение `exclude` из rotation и передачу дальше.

### 5. Место вызова `rotation.add()`: после успешного показа

**Выбор:** В хэндлерах, после того как рецепт успешно получен из GitHub и готов к отправке.

**Обоснование:** Если GitHub вернул 404 и рецепт был пропущен — он не должен попасть в rotation. Только фактически показанные рецепты исключаются.

## Risks / Trade-offs

- **[Данные теряются при рестарте]** → Приемлемо. Один повторный рецепт после рестарта незаметен.
- **[Deque может содержать slug'и удалённых рецептов]** → Безопасно: фильтрация использует set intersection, несуществующие slug'и просто не совпадут с текущим списком.
- **[Потребление памяти]** → Минимально: ~100 байт на пользователя × категорию. Даже при 10K пользователей и 3 категориях — ~3 MB.
