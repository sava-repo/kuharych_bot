# Proposal: Расчёт КБЖУ на порцию блюда

## Why

Рецепты из рилсов хранят только `title/ingredients/steps/category/tags`. Пользователь видит состав, но не понимает энергетическую ценность: чтобы прикинуть калории и БЖУ, приходится открывать сторонний счётчик и вбивать каждый ингредиент вручную. Для кулинарного бота это разрыв в базовом сценарии «оцени, подходит ли блюдо под твою норму».

Существующая архитектура хорошо располагает к решению: LLM и так парсит рецепт в одном вызове и владеет примерными знаниями о составе продуктов. Достаточно расширить промпт и модель — без новых внешних зависимостей, без миграции БД и без правок пайплайна/хендлеров.

## What Changes

- **Промпт извлечения** (`prompts/system.txt`) дополнен блоком КБЖУ: LLM возвращает `Порции` и тотальные `Калории/Белки/Жиры/Углеводы` на всё блюдо.
- **`Recipe` (dataclass)** получает 5 числовых полей (`portions`, `calories`, `protein`, `fat`, `carbs` — тотальные); значения сериализуются в YAML frontmatter `content_md` (по образцу `tags`), **без колонок в БД**.
- **Расчёт «на порцию»** выполняется кодом в `Recipe.format_message`: `total / portions`, с округлением и маркером оценки `≈`. Единая точка рендера покрывает все 4 места показа (link, menu ×2, buttons).
- **Парсер** (`services/recipe_parser._parse_recipe_response`) разбирает 5 новых строк; при отсутствии/нечисловых значениях поле остаётся `0` → строка КБЖУ не выводится (плавная деградация).
- **Бэкфилл существующих рецептов**: новый модуль `services/nutrition.py` + отдельный промпт `prompts/nutrition.txt` (только КБЖУ, без переизвлечения рецепта) + runnable `python -m services.nutrition_backfill`.

## Capabilities

### Added Capabilities
- `recipe-nutrition`: оценка КБЖУ рецепта (LLM) и расчёт значений на порцию при отображении.

## Impact

- **Промпты**: `prompts/system.txt` (+блок КБЖУ), `prompts/nutrition.txt` (новый, для бэкфилла).
- **Модели**: `models/recipe.py` (+5 полей, `format_message` с делением и `≈`, `to_markdown`/`from_markdown` с frontmatter).
- **Парсер**: `services/recipe_parser.py` (парсинг 5 строк, толерантность к пропускам).
- **Новые сервисы**: `services/nutrition.py` (`estimate_nutrition`, `backfill_all`), `services/nutrition_backfill.py` (точка входа `__main__`).
- **Не затрагивается**: `services/recipe_pipeline.py`, `services/group_manager.py`, `services/database.py`, `handlers/*`, схема БД, `requirements.txt`.
- **Тесты**: `tests/unit/test_models.py`, `tests/unit/test_parser.py`, новый `tests/unit/test_nutrition.py`.

## Non-goals

- Точные КБЖУ из внешней БД продуктов (Edamam/FatSecret/FoodRepo) — LLM-оценки достаточно для «прикинуть калорийность»; интеграция food-БД с маппингом русских названий — отдельный продукт.
- Редактируемые порции пользователем (пересчёт по своему числу) — `portions` определяет LLM; пользовательская правка отложена.
- Фильтрация/сортировка рецептов по КБЖУ (потребовало бы колонок в БД) — хранение остаётся в frontmatter.
- КБЖУ для GitHub-рецептов (`format_markdown_for_chat`) — вне MVP; отображение статического markdown не меняется.
- Дашборд: показ КБЖУ на страницах Streamlit — может быть добавлен попутно, но не является обязательным.

## Точность и доверие

LLM-оценка КБЖУ носит ориентировочный характер (±20–40%). Это подчёркивается маркером `≈` в выводе и числом порций в скобках `(1 порция из 4)`, чтобы пользователь мог сам сверить адекватность. Бот не претендует на роль медицинского/диетического трекера.
