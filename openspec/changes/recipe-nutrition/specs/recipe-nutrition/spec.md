# Spec Delta: recipe-nutrition

> Новая capability: оценка КБЖУ рецепта (LLM) и расчёт значений на порцию при отображении.

## ADDED Requirements

### Requirement: Оценка КБЖУ при извлечении рецепта
При извлечении нового рецепта из видео система SHALL передавать LLM запрос на оценку КБЖУ в том же вызове, что извлекает название, ингредиенты и шаги. LLM SHALL вернуть тотальные значения (калории, белки, жиры, углеводы) на всё блюдо и число порций.

#### Scenario: Новый рецепт получает КБЖУ
- **WHEN** пользователь добавляет ранее неизвестный рилс
- **THEN** LLM возвращает порции и тотальные калории/белки/жиры/углеводы
- **AND** эти значения сохраняются в рецепте

#### Scenario: LLM не дал численных значений
- **WHEN** LLM не вернул строки КБЖУ или вернул не-числовые значения
- **THEN** соответствующие поля рецепта остаются равными 0
- **AND** отображение рецепта не ломается (строка КБЖУ не выводится)

### Requirement: Хранение КБЖУ в контенте рецепта
Система SHALL хранить КБЖУ (portions, calories, protein, fat, carbs — тотальные) в YAML frontmatter `content_md`, по образцу существующего хранения `tags`. Схема базы данных НЕ ДОЛЖНА (MUST NOT) изменяться: КБЖУ не является колонкой таблицы `recipes`.

#### Scenario: Сериализация в Markdown
- **WHEN** рецепт сохраняется в `content_md`
- **THEN** frontmatter содержит строки `portions`, `calories`, `protein`, `fat`, `carbs` с тотальными целыми значениями

#### Scenario: Чтение старого рецепта без КБЖУ
- **WHEN** загружается рецепт, в frontmatter которого нет блока КБЖУ
- **THEN** поля КБЖУ принимают значения по умолчанию (0)
- **AND** ошибки парсинга не возникает

### Requirement: Расчёт КБЖУ на порцию при отображении
При формировании сообщения с рецептом система SHALL вычислять значения на одну порцию как `total / portions` (целочисленное деление). Строка КБЖУ SHALL отображаться под названием только при `calories > 0` и `portions > 0`.

#### Scenario: Вывод КБЖУ на порцию
- **GIVEN** рецепт с `calories=1280`, `protein=80`, `fat=40`, `carbs=120`, `portions=4`
- **WHEN** рецепт отображается в чате
- **THEN** под названием выводится строка вида `≈320 ккал · Б 20 / Ж 10 / У 30   (1 порция из 4)`
- **AND** символ `≈` присутствует как индикатор оценочного характера значений

#### Scenario: Блюдо на одну порцию
- **GIVEN** рецепт с `portions=1`
- **WHEN** рецепт отображается в чате
- **THEN** в скобках указано `(на 1 порцию)`

#### Scenario: Рецепт без КБЖУ
- **GIVEN** рецепт с `calories=0` (например, не прошедший бэкфилл)
- **WHEN** рецепт отображается в чате
- **THEN** строка КБЖУ не выводится
- **AND** остальной вывод (название, ингредиенты, шаги, категория) не меняется

#### Scenario: Единый рендер во всех точках показа
- **WHEN** рецепт показывается при сохранении, при случайной выдаче, при поиске или при восстановлении сообщения
- **THEN** строка КБЖУ формируется одним и тем же методом `Recipe.format_message`

### Requirement: Бэкфилл КБЖУ существующих рецептов
Система SHALL предоставлять утилиту для расчёта КБЖУ уже сохранённых рецептов через отдельный LLM-промпт, не переизвлекая название, ингредиенты и шаги. Бэкфилл SHALL запускаться вручную как runnable module и обновлять только `content_md` (значения КБЖУ), не затрагивая `recipe_ingredients`, `reel_index`, `group_recipes`.

#### Scenario: Бэкфилл рецепта без КБЖУ
- **GIVEN** в БД есть рецепт без блока КБЖУ в `content_md`
- **WHEN** запускается утилита бэкфилла
- **THEN** для рецепта вызывается только-КБЖУ промпт с title + ingredients + steps
- **AND** полученные значения записываются в `content_md` рецепта
- **AND** title/ingredients/steps остаются без изменений

#### Scenario: Бэкфилл пропускает уже имеющие КБЖУ
- **GIVEN** рецепт уже содержит ненулевые `calories` в `content_md`
- **WHEN** запускается бэкфилл
- **THEN** этот рецепт пропускается (повторный LLM-вызов не выполняется)

#### Scenario: Сухой прогон бэкфилла
- **WHEN** бэкфилл запускается с флагом dry-run
- **THEN** LLM-вызовы выполняются (или только отбираются кандидаты)
- **AND** изменения в БД не записываются
- **AND** формируется отчёт о том, какие рецепты были бы обновлены
