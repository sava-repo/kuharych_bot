# Design: Расчёт КБЖУ на порцию блюда

## Контекст и решения explore-режима

Источник КБЖУ — **LLM-оценка в текущем вызове** парсера (без второго запроса и без внешних food-БД). Порции определяет LLM из объёма ингредиентов и контекста. Хранение — в YAML frontmatter `content_md` (как `tags`), **миграция БД не нужна**. Отображение — всегда строкой под названием, единая точка `Recipe.format_message`. Существующие рецепты бэкфиллятся отдельным только-КБЖУ промптом через `python -m services.nutrition_backfill`. Оценка маркируется символом `≈`.

## Модель данных: total → per-portion

LLM возвращает **тотальные** значения на всё блюдо + число порций. Код делит `total / portions` при отображении. Это даёт:
- буквальный «расчёт на порцию» (делает код, а не LLM);
- задел под будущие редактируемые порции (пересчёт из total тривиален);
- одно место арифметики — `format_message`.

```
  LLM-ответ                Recipe (dataclass, total)      format_message (per-portion)
  ───────────────          ──────────────────────────     ───────────────────────────────
  Порции: 4                portions = 4                   ≈320 ккал · Б 20 / Ж 10 / У 30
  Калории: 1280            calories = 1280                  (1 порция из 4)
  Белки: 80                protein = 80
  Жиры: 40                 fat = 40
  Углеводы: 120            carbs = 120
```

### Почему целые числа

Все 5 полей — `int`. Калории и граммы Б/Ж/У в бытовом контексте показываются целыми; LLM-оценка ±20–40% делает десятичные доли бессмысленным шумом. Парсинг проще (одно `int(...)` с защитой), сериализация в frontmatter чище.

### Деградация

Если LLM не вернул КБЖУ или верёл не-число → поле `0`. Строка КБЖУ выводится **только при `calories > 0`** (и `portions > 0`). Таким образом старые рецепты до бэкфилла, а также любой сбой LLM, не ломают отображение — пользователь просто видит рецепт без КБЖУ.

## Промпт извлечения (`prompts/system.txt`)

К существующему формату ответа добавляется хвост:

```
Порции: [целое число — на сколько порций рассчитано блюдо]
Калории: [целое — суммарно на всё блюдо]
Белки: [целое, граммы — суммарно]
Жиры: [целое, граммы — суммарно]
Углеводы: [целое, граммы — суммарно]
```

Шаблоны `{categories}` / `{default}` сохраняются. Значения — ориентировочные, LLM оценивает по известному составу продуктов. Порядок: блок КБЖУ идёт после `Категория:`.

## Модель `Recipe`

```python
@dataclass
class Recipe:
    title: str
    ingredients: list[str]
    steps: list[str]
    source: str
    category: str = ""
    tags: list[str] = field(default_factory=list)
    # КБЖУ — тотальные значения на всё блюдо (LLM-оценка)
    portions: int = 0
    calories: int = 0
    protein: int = 0
    fat: int = 0
    carbs: int = 0
```

### `format_message`

После `🍳 {title}` и пустой строки, **если `calories > 0` и `portions > 0`**, вставляется:

```
≈{calories//portions} ккал · Б {protein//portions} / Ж {fat//portions} / У {carbs//portions}   (1 порция из {portions})
```

При `portions == 1` — `(на 1 порцию)`. Целочисленное деление `//` (округление вниз) — соответствует «оценке». Дальше как раньше: Ингредиенты, Приготовление, Категория.

### `to_markdown` / `from_markdown`

`to_markdown` дописывает в YAML frontmatter (после `tags`):

```
portions: 4
calories: 1280
protein: 80
fat: 40
carbs: 120
```

`from_markdown` парсит эти строки (как парсит `title:`); при отсутствии — поля остаются `0`. Старые рецепты (без блока КБЖУ) читаются без ошибок: неизвестные строки frontmatter и так пропускаются существующей логикой.

> Примечание: существующий `from_markdown` содержит ветку `if stripped.startswith(("category:", "source:", "created:", "tags:")): continue` — её нужно дополнить парсингом `portions/calories/protein/fat/carbs:` и оставить `continue` для остальных пропускаемых.

## Парсер (`services/recipe_parser._parse_recipe_response`)

Добавляется извлечение 5 значений из текста ответа (по образцу извлечения `Категория:`):

- для каждой строки вида `Порции: 4` / `Калории: 1280` / … берётся часть после `:`, `strip`, попытка `int(...)`;
- при `ValueError` (нечисло, пусто) → поле остаётся `0`;
- отсутствие строк целиком → поля `0` (деградация).

Значения прокидываются в конструктор `Recipe(...)`. Категория и теги парсятся как прежде.

## Бэкфилл существующих рецептов

**Почему отдельный промпт:** гнать старый рецепт через полный `system.txt` значит переизвлечь `title/ingredients/steps` и переписать контент — риск регрессии. Поэтому для бэкфилла используется **только-КБЖУ промпт** `prompts/nutrition.txt`: на вход подаётся `title + ingredients + steps` уже распознанного рецепта, на выходе — только `Порции/Калории/Белки/Жиры/Углеводы`.

### `services/nutrition.py`

```python
async def estimate_nutrition(recipe: Recipe) -> NutritionEstimate:
    """Только-КБЖУ LLM-вызов по prompts/nutrition.txt.
    Возвращает dataclass NutritionEstimate(portions, calories, protein, fat, carbs)."""

def backfill_all(*, dry_run: bool = False, limit: int | None = None) -> BackfillReport:
    """Итерирует рецепты без calories; для каждого — estimate_nutrition,
    патчит content_md (to_markdown с новыми полями) и UPDATE в recipes.
    Логирует прогресс; при dry_run только отчёт без записи."""
```

- Отбор кандидатов: `SELECT recipe_id, content_md FROM recipes` в Python-цикле, `Recipe.from_markdown` → `if recipe.calories == 0`.
- Обновление: пересобрать `content_md` через `recipe.to_markdown(created=...)` с подставленными значениями и `UPDATE recipes SET content_md = ? WHERE recipe_id = ?`.
- Не трогает `recipe_ingredients`, `reel_index`, `group_recipes` — КБЖУ живёт только в `content_md`.

### `services/nutrition_backfill.py`

```python
# python -m services.nutrition_backfill [--dry-run] [--limit N]
if __name__ == "__main__":
    # парсинг argv, asyncio.run(backfill_all(...)), печать отчёта
```

Идиома `python -m services.nutrition_backfill` (как `python -m pytest`) — без нового скрипта в корне, переиспользует сервисный слой.

## Отображение в чате

```
🍳 Борщ

≈320 ккал · Б 20 / Ж 10 / У 30   (1 порция из 4)

📋 Ингредиенты:
• 400 г говядины
...

👨‍🍳 Приготовление:
1. ...

📂 Категория: основное блюдо
```

Единый рендер `format_message` покрывает все точки вызова:
- `handlers/link.py:138, 148`
- `handlers/menu.py:138, 283, 309`
- `handlers/buttons.py:52`

Правок в хендлерах **нет** — КБЖУ «проявляется» автоматически.

## Что НЕ меняется

| Компонент | Почему |
|---|---|
| `services/recipe_pipeline.py` | КБЖУ — поле `Recipe`; пайплайн пробрасывает его как есть |
| `services/group_manager.py` | `save_recipe` пишет `content_md`, в котором уже КБЖУ; индексация по ингредиентам не затронута |
| `services/database.py` / схема | КБЖУ в frontmatter `content_md`, колонок не добавляем |
| `handlers/*` | все вызывают `format_message` |
| `requirements.txt` | новых зависимостей нет |

## Риски и митигации

| Риск | Митигация |
|---|---|
| LLM-оценка неточна (±20–40%) | Маркер `≈` + число порций в скобках; отказ от претензии на медицинскую точность (в Non-goals) |
| LLM не вернул или верёл мусор вместо чисел | Толерантный парсер → поля `0` → строка КБЖУ не выводится; рецепт показывается как раньше |
| Бэкфилл тратит токены/время на большой БД | `--limit`, `--dry-run`, построчный логинг; запуск вручную в удобное время |
| Бэкфилл ломает существующий `content_md` | Только-КБЖУ промпт не переизвлекает рецепт; патч через `to_markdown` сохраняет все поля; `dry_run` для сверки |
| `portions == 0` от LLM (не смог оценить) | Деление пропускается, строка КБЖУ не выводится (инвариант `portions > 0`) |
| Дубли/расхождение КБЖУ у одного рецепта в разных группах | КБЖУ хранится в `recipes.content_md` (один `recipe_id`), одинаков для всех групп — по аналогии с `title`/`steps` |

## Решения

### Почему КБЖУ тотальные, а не per-portion
LLM надёжнее оценивает «всё блюдо из 400 г курицы + 200 г риса», чем «одна порция» (где нужно сначала угадать порции и поделить). Хранение total + portions даёт честный «расчёт» кодом и задел под редактируемые порции.

### Почему хранение в frontmatter, а не колонки БД
КБЖУ — атрибут контента рецепта (как `tags`), а не запросный фильтр. Колонки потребовали бы миграции `recipes` и не несут ценности без фильтрации (которая в Non-goals). Frontmatter читается `from_markdown` без правок схемы.

### Почему бэкфилл отдельным промптом
Повторный полный вызов `system.txt` переизвлечёт title/ingredients/steps и перепишет контент рецепта — регрессия. Только-КБЖУ промпт принимает уже распарсенные ингредиенты и возвращает 5 чисел, не трогая остальное.

### Почему runnable module, а не админ-команда
В боте нет админ-команд; бэкфилл — разовая операция. `python -m services.nutrition_backfill` следует идиомам запуска (как `pytest`), переиспользует сервисный слой и не плодит корневые скрипты.
