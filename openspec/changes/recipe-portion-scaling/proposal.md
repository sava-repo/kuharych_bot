# Proposal: Пересчёт ингредиентов по числу порций

## Why

У рецепта уже есть базовое число порций (`portions`, оценка LLM из change `recipe-nutrition`), но ингредиенты показываются ровно в исходном виде — на фиксированное число порций. Чтобы приготовить больше/меньше, пользователь пересчитывает «в уме» (400 г на 4 порции → сколько на 6?). Для кулинарного бота это трение в базовом сценарии «готовлю на свою семью».

Хочется под рецептом кнопки `−1 порция / +1 порция`, которые мгновенно пересчитывают количества ингредиентов под выбранное число порций. Способ приготовления не меняется. Результат пересчёта эфемерен — живёт только в текущем сообщении и не пишется в БД.

## What Changes

- **Парсер количества** (`services/ingredient_scaler.py`): regex-разбор ведущего числа (целое / десятичное `1,5` / дробь `1/2`, `1 1/2` / диапазон `3-4`); масштабирование через `fractions.Fraction` на коэффициент `target/base`; ингредиенты без числа («по вкусу соль») — без изменений.
- **`Recipe.format_message`**: опциональный `portions_override: int | None` — при переданном значении ингредиенты масштабируются; способ приготовления и КБЖУ-строка остаются на базовых значениях. Единая точка рендера.
- **Клавиатура порций** (`handlers/keyboards.py`): ряд `[➖][N порций][➕]`, `callback_data = psc:{rk}:{target}`. Скрыты на границах 1..20. Не показываются при `base_portions == 0`.
- **Хендлер `psc:`** (`handlers/buttons.py`): по `rk` читает `recipe_id/group_id`, коэффициент `target/recipe.portions`, рендер через `format_message(portions_override=target)`. Stateless — состояние в `callback_data`, кэш не мутируется.
- **Точки подключения**: случайный рецепт и `/open` (`menu.py`), восстановление вида и после переноса (`buttons.py`). Экран сохранения дубликата (`link.py`) — без кнопок.

## Capabilities

### Added Capabilities
- `recipe-portion-scaling`: пересчёт ингредиентов рецепта на произвольное число порций (in-memory, без записи в БД).

## Impact

- **Новые сервисы**: `services/ingredient_scaler.py`.
- **Модели**: `models/recipe.py` (`format_message` + `portions_override`).
- **Хендлеры**: `handlers/buttons.py` (+`psc:`), `handlers/menu.py` (клавиатура), `handlers/keyboards.py` (+`portions_keyboard` / расширение `recipe_keyboard`).
- **Не затрагивается**: схема БД, `recipe_pipeline.py`, `group_manager.py`, `recipe_parser.py`, промпты, `requirements.txt`.
- **Тесты**: новый `tests/unit/test_ingredient_scaler.py`, дополнения в `test_models.py`, `test_buttons.py`.

## Non-goals

- Сохранение выбранного числа порций между сессиями / в БД — пересчёт эфемерен.
- Масштабирование КБЖУ — строка остаётся на базовых значениях (принятая небольшая визуальная несогласованность подписи).
- LLM-пересчёт — regex достаточно; LLM на каждый клик слишком медленный.
- Пересчёт для GitHub-рецептов (`format_markdown_for_chat`) и дашборда Streamlit — вне MVP.
- Структурированное хранение ингредиентов (`{qty, unit, name}`) — отдельная миграция.
