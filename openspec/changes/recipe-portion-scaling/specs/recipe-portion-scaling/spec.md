# Spec Delta: recipe-portion-scaling

> Новая capability: пересчёт ингредиентов рецепта на произвольное число порций (in-memory, без записи в БД).

## ADDED Requirements

### Requirement: Под рецептом выводятся кнопки пересчёта порций
При отображении рецепта с `portions > 0` система SHALL показывать под рецептом ряд inline-кнопок `[−1 порция][N порций][+1 порция]`. Кнопка уменьшения SHALL быть скрыта при текущем значении 1, кнопка увеличения — при 20.

#### Scenario: Первичный показ рецепта с портами
- **GIVEN** рецепт с `portions=4`
- **WHEN** рецепт отображается (случайная выдача, `/open`, восстановление вида)
- **THEN** под рецептом выводится ряд `[-1 порция][4 порции][+1 порция]`

#### Scenario: Рецепт без порций — кнопок нет
- **GIVEN** рецепт с `portions=0`
- **WHEN** рецепт отображается в чате
- **THEN** ряд порций не выводится
- **AND** остальные кнопки (удаление/перенос/источник) остаются

#### Scenario: Границы диапазона
- **GIVEN** текущее число порций равно 1
- **WHEN** рецепт отображается
- **THEN** кнопка уменьшения (`-1 порция`) скрыта
- **AND** кнопка увеличения присутствует

### Requirement: Кнопки пересчитывают ингредиенты, но не способ приготовления
При нажатии кнопки `−1`/`+1` система SHALL пересчитать количества ингредиентов на выбранное число порций через коэффициент `target_portions / base_portions`. Способ приготовления (шаги) SHALL остаться без изменений. Строка КБЖУ SHALL остаться на базовых значениях (без пересчёта).

#### Scenario: Увеличение порций масштабирует ингредиенты
- **GIVEN** рецепт с `portions=4` и ингредиентом «400 г говядины»
- **WHEN** пользователь нажимает `+1 порция` (target=5)
- **THEN** сообщение обновляется: ингредиент «500 г говядины»
- **AND** шаги приготовления не меняются
- **AND** ряд кнопок отражает новое текущее значение (`5 порций`)

#### Scenario: Уменьшение порций
- **GIVEN** рецепт с `portions=4` и ингредиентом «400 г говядины»
- **WHEN** пользователь нажимает `-1 порция` (target=3)
- **THEN** ингредиент пересчитывается в «300 г говядины»

#### Scenario: КБЖУ не пересчитывается
- **GIVEN** рецепт с `calories=1280`, `portions=4` (базовые)
- **WHEN** пользователь меняет число порций кнопками
- **THEN** строка КБЖУ остаётся на базовых значениях (`1 порция из 4`)

### Requirement: Ингредиенты без числа остаются без изменений
При пересчёте система SHALL масштабировать только ведущее число в строке ингредиента (целое, десятичное, дробь, смешанная дробь, диапазон). Ингредиенты без ведущего числа («по вкусу», «щепотка») SHALL выводиться как есть.

#### Scenario: Ингредиент без количества
- **GIVEN** ингредиент «по вкусу соль»
- **WHEN** выполняется пересчёт на любое число порций
- **THEN** строка остаётся «по вкусу соль»

#### Scenario: Дробное количество
- **GIVEN** ингредиент «1/2 ч. л. соли» и пересчёт с коэффициентом 3/2
- **THEN** строка становится «0,75 ч. л. соли»

### Requirement: Результат пересчёта эфемерен
Результат пересчёта SHALL существовать только в текущем сообщении и НЕ ДОЛЖЕН (MUST NOT) записываться в базу данных. Состояние текущего числа порций SHALL кодироваться в `callback_data` кнопок; кэш рецептов SHALL использоваться только для получения `recipe_id`/`group_id`, без мутации.

#### Scenario: Повторное открытие сбрасывает число порций
- **GIVEN** пользователь менял число порций до 7 в одном сообщении
- **WHEN** пользователь открывает тот же рецепт заново (новый `/open`)
- **THEN** число порций сбрасывается к базовому (`recipe.portions`)
- **AND** ингредиенты показываются в исходном виде

#### Scenario: Подделанное значение в callback
- **WHEN** в `callback_data` пришло `target` вне диапазона 1..20
- **THEN** система SHALL ограничить (clamp) значение до границ 1..20 перед рендером
